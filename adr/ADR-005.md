# ADR-002: Single Event Dispatcher — funnel + topic pattern matching

Date: 2026-01-07
Status: Accepted
Deciders: Farm Node Team
Authors: Gaia FarmNode team (documented by an engineer)

## Context

- The Farm Node produces a variety of local events (telemetry from devices, local alerts, device status, etc.) that may be forwarded to the Hub depending on policy and need.
- There are two sensible approaches:
  - Many per-topic dispatchers (one GenServer per topic), or
  - A single dispatcher that receives all (or a small set of) topics and uses pattern matching on topic names to decide what to do.
- The system must respect project rules:
  - Data must not be pushed to the Hub unless allowed by the local `DataSharingPolicy`.
  - Provisioning and mTLS are required for Hub communication (handled elsewhere in `HubConnection`).
  - Faults in sensor drivers or other hardware must not crash core logic; the supervision tree must isolate failures.
- We expect moderate event volumes now (low-to-moderate throughput per topic), and we prefer to start with a simple and auditable implementation while keeping a clear migration path to per-topic workers if load or failure isolation requirements change.

## Decision

We will implement a single `Gaia.FarmNode.HubConnection.EventDispatcher` process (a single dispatcher), which:

- Subscribes to a small set of ingress topics (e.g. `telemetry:all`, `local_alerts`, and any other broad topics we deem necessary).
- Internally uses topic-name pattern matching to route events to logical buckets (e.g. `"telemetry:*"` vs `"local_alerts"`).
- Maintains small per-pattern buffers (buckets) inside the single process rather than a single monolithic queue. Each bucket is flushed independently according to its own policy (configurable buffer size and/or time interval).
- On flush, applies the `DataSharingPolicy` checks appropriate for the topic/pattern before sending to the Hub via an injectable `HubClient` behaviour.
- Exposes configuration for buffer sizes, flush intervals, and which topic patterns are considered for forwarding.

Rationale:
- This meets the current needs: simpler implementation, centralized policy enforcement, and easier observability.
- Per-pattern buckets inside the single dispatcher reduce the chance that a single hot-topic dominates memory growth or flush behavior while still keeping a single process entry point (simpler to debug).
- If the dispatcher becomes a bottleneck in the future, we can migrate to per-topic workers behind a `DynamicSupervisor` — the migration path is straightforward because the dispatch logic and bucket semantics will be well defined and contained.

## Consequences

Pros
- Simpler code and centralized place to enforce policies such as `DataSharingPolicy`.
- Easier to add global rate-limits and global monitoring.
- Less process supervision overhead (fewer processes than per-topic solution).
- Simpler to test: one component to test the pattern logic, buffering, policy enforcement, and Hub client integration.

Cons / Risks
- Single process means potential contention; if event throughput increases, mailbox or CPU on that process can become a bottleneck.
- A crash of the single dispatcher impacts multiple topics until it is restarted and buffers repopulated.
- The single process must carefully manage memory (buckets must be bounded and/or persist to disk if required).

Mitigations
- Keep small, bounded per-pattern buffers; drop or persist old events if necessary and log metrics.
- Add monitoring: per-bucket queue depth, flush latency, mailbox length, flush success/failure counts.
- Define thresholds / alarms and a clear rollback / migration path to per-topic workers.
- Ensure the dispatcher's supervision node has appropriate restart limits (`:max_restarts/:max_seconds`).

## Implementation notes

API & message shape
- Standardize in-process event message shape to: `{:telemetry, topic :: String.t(), payload :: map()}`.
  - (Action item: update `Gaia.FarmNode.EventStream` and consumers to use/expect this message shape.)
- The dispatcher will `EventStream.subscribe/1` to the set of topics we want to consider (e.g., `"telemetry:all"`, `"local_alerts"`). If `EventStream` later supports pattern subscriptions we can simplify subscription.

Buffering & flushing
- Dispatcher maintains `state.buckets :: %{bucket_name => %{buffer: [event], size: integer}}`.
- Config (suggested keys under `:farm_node, :event_dispatcher`):
  - `buffer_size` (default 10) — flush when bucket size >= this
  - `flush_interval_ms` (default 60_000) — periodic flush
  - `topic_patterns` — list of regex/glob patterns and routing metadata for each pattern
  - `flush_retry` — strategy for failed pushes (e.g., `{:exponential_backoff, max_retries}`)
- Flushing policy:
  - On buffer threshold reached → synchronous attempt to push that bucket (async to Hub client as needed).
  - On periodic timer tick → attempt to flush all non-empty buckets.
  - On `terminate/2` → attempt a final synchronous flush of all buckets.

Pattern matching (example)
- Dispatcher inspects `topic` and groups it, e.g.:
```/dev/null/event_dispatcher_pattern.ex#L1-40
def handle_info({:telemetry, topic, payload}, state) do
  case topic do
    "telemetry:" <> _rest -> add_to_bucket(:telemetry, {topic, payload}, state)
    "local_alerts" -> add_to_bucket(:alerts, {topic, payload}, state)
    _ -> {:noreply, state} # ignore or log
  end
end
```

Hub & Policy integration
- Before sending events to the Hub, the dispatcher will ask `Gaia.FarmNode.DataSharingPolicy` whether the event/topic is allowed to be shared. For example:
  - `if DataSharingPolicy.allow?(topic, event) do HubClient.push(events) end`
- Hub calls must use provisioned mTLS credentials; `HubClient` is a behaviour with a test fake implementation.
- If a push fails, the dispatcher will apply a retry/backoff strategy per-bucket and emit metrics/logging.

Configuration sketch
```/dev/null/config.exs#L1-40
config :farm_node, :event_dispatcher,
  buffer_size: 10,
  flush_interval_ms: 60_000,
  topic_patterns: [
    %{pattern: "^telemetry:", bucket: :telemetry, share_policy: :share_telemetry},
    %{pattern: "^local_alerts$", bucket: :alerts, share_policy: :share_local_alerts}
  ],
  flush_retry: %{strategy: :exponential_backoff, max_retries: 5}
```

Observability & testing
- Emit metrics (Prometheus / Telemetry):
  - `event_dispatcher.bucket_depth` (labels: bucket)
  - `event_dispatcher.flush_success` / `event_dispatcher.flush_failure`
  - `event_dispatcher.flush_duration`
- Unit tests:
  - Topic → bucket mapping logic.
  - Buffering behavior and flush-on-threshold.
  - Periodic flush behavior.
  - Integration test with fake `HubClient` and fake `DataSharingPolicy` verifying correct gating.
- Integration tests:
  - Simulate Hub failures and assert retry behavior and metrics.

Migration & rollback plan
- If dispatcher becomes overloaded, shift to per-topic workers:
  1. Add a `DynamicSupervisor` and per-topic `DispatcherWorker` processes.
  2. Move bucket flushing and buffering into workers; keep existing dispatcher as a router that starts workers on demand (backwards compatible).
  3. Gradual rollout: start workers for hottest patterns first.
- Rollback: revert to the single-dispatcher branch if issues appear, add more observability and back-pressure.

When to revisit this decision
- If we see sustained dispatcher CPU > 70% or 95th percentile event-to-hub latency > 500ms.
- If the dispatcher’s mailbox length stays > N (configurable) for > 1 minute.
- If we need strict per-topic QoS or significantly different retry/persistence models per topic.

## Alternatives considered

- Per-topic dispatchers (one worker per topic)
  - Pros: isolation, per-topic tuning, natural scaling via processes.
  - Cons: more supervision complexity and resource overhead; more moving parts to reason about initially.

- Macro-generated per-topic modules (`use`/`defmacro`)
  - Pros: compile-time convenience for a small fixed set of topics.
  - Cons: less flexible for runtime topic configuration; macros make debugging/testing slightly harder.

We considered both and picked the single dispatcher (with per-pattern buckets inside) for the reasons above (simplicity and centralized policy enforcement), while keeping a clear migration path to per-topic workers.
